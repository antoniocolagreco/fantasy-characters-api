# Database Index Optimization Strategy

## Overview

This document outlines the comprehensive database index optimization strategy implemented for the
Fantasy Characters API. The optimizations are based on actual query patterns analysis from the
codebase and are designed to improve performance for the most common database operations.

## Index Strategy Principles

### 1. Query Pattern Analysis

- Analyzed actual SQL queries generated by Prisma ORM
- Identified frequent filter combinations from service layer code
- Examined RBAC (Role-Based Access Control) query patterns
- Reviewed statistics and time-based query requirements

### 2. Index Design Principles

- **Selectivity First**: Most selective columns placed first in composite indexes
- **Query Coverage**: Indexes designed to cover complete WHERE clauses
- **Sort Optimization**: Included ORDER BY columns in indexes
- **Partial Indexes**: Used WHERE clauses for frequently filtered subsets
- **Text Search**: GIN indexes with trigrams for fuzzy text search

## Optimization Categories

### 1. User Management Optimizations

#### Primary Query Patterns

- Authentication: `email + isActive + isEmailVerified`
- User filtering: `role + isActive + createdAt`
- Activity tracking: `lastLogin` descending
- Search: fuzzy name matching
- Statistics: time-based `createdAt` ranges

#### Indexes Added

```sql
-- Composite authentication index
idx_users_email_status (email, isActive, isEmailVerified)

-- User filtering and pagination
idx_users_role_active_created (role, isActive, createdAt DESC)

-- Time-based statistics
idx_users_created_at_desc (createdAt DESC)
idx_users_last_login (lastLogin DESC)

-- Fuzzy search capability
idx_users_name_trgm (name gin_trgm_ops)
```

### 2. Session Management Optimizations

#### Primary Query Patterns

- Token validation: `token + isRevoked + expiresAt`
- User session lookup: `userId + isRevoked + expiresAt`
- Cleanup operations: `expiresAt` for expired tokens

#### Indexes Added

```sql
-- Active token lookups
idx_refresh_tokens_token_active (token, isRevoked, expiresAt)
idx_refresh_tokens_user_active (userId, isRevoked, expiresAt)
```

### 3. Content Management (RBAC Pattern)

#### RBAC Query Pattern

All content models (Images, Tags, Skills, Perks, Races, Archetypes, Items, Characters) follow this
pattern:

```typescript
// Non-authenticated: only PUBLIC
{
  visibility: 'PUBLIC'
}

// Regular users: PUBLIC + own PRIVATE
{
  OR: [{ visibility: 'PUBLIC' }, { ownerId: userId, visibility: 'PRIVATE' }]
}

// Admins/Moderators: everything
{
}
```

#### Indexes Added for Each Content Model

```sql
-- RBAC filtering optimization
idx_[model]_owner_visibility (ownerId, visibility, [sort_field])

-- Public content fast access
idx_[model]_visibility_created (visibility, createdAt DESC)
```

### 4. Game System Specific Optimizations

#### Skills & Perks

- Level-based filtering: `requiredLevel` ranges
- Search by name with fuzzy matching

```sql
idx_skills_level_visibility (requiredLevel, visibility, ownerId)
idx_perks_level_visibility (requiredLevel, visibility, ownerId)
```

#### Items

- Complex filtering: `rarity + slot + requiredLevel + visibility`
- Combat stats queries: `damage + defense`

```sql
idx_items_rarity_slot_level (rarity, slot, requiredLevel, visibility)
idx_items_combat_stats (damage, defense)
idx_items_slot_level (slot, requiredLevel)
```

#### Characters

- Multi-criteria filtering: `ownerId + visibility + level + createdAt`
- Race/Archetype combinations: `raceId + archetypeId + visibility`
- Stat-based queries: `level + experience + strength + constitution`

```sql
idx_characters_owner_visibility (ownerId, visibility, level DESC, createdAt DESC)
idx_characters_race_archetype (raceId, archetypeId, visibility)
idx_characters_stats (level, experience, strength, constitution)
```

## Text Search Optimization

### Trigram Indexes (GIN)

All models with searchable name fields include trigram indexes for fuzzy search:

```sql
CREATE INDEX idx_[model]_name_trgm ON [table] USING gin (name gin_trgm_ops);
```

Benefits:

- Enables fuzzy text search with `ILIKE '%term%'`
- Supports typo-tolerant search
- Faster than full table scans for text matching

## Performance Impact Analysis

### Expected Improvements

1. **User Authentication**: 50-80% faster login queries
2. **Content Filtering**: 60-90% faster RBAC queries
3. **Search Operations**: 70-95% faster text search
4. **Statistics Queries**: 80-95% faster time-based aggregations
5. **Complex Filtering**: 40-70% faster multi-criteria queries

### Memory Usage

- Estimated additional index storage: 15-25% of table size
- Memory benefit from reduced query execution time
- Better query plan caching

### Maintenance Considerations

- Indexes automatically maintained by PostgreSQL
- Write operations slightly slower (marginal impact)
- Periodic `ANALYZE` recommended for optimal query plans

## Query Examples Optimized

### 1. User Authentication

```sql
-- Before: Table scan
SELECT * FROM users WHERE email = ? AND isActive = true AND isEmailVerified = true;

-- After: Index seek using idx_users_email_status
```

### 2. Content RBAC Filtering

```sql
-- Before: Multiple table scans
SELECT * FROM skills
WHERE (visibility = 'PUBLIC' OR (ownerId = ? AND visibility = 'PRIVATE'))
ORDER BY name;

-- After: Index seek using idx_skills_owner_visibility
```

### 3. Character Building Queries

```sql
-- Before: Multiple index lookups
SELECT * FROM characters
WHERE raceId = ? AND archetypeId = ? AND visibility = 'PUBLIC'
ORDER BY level DESC;

-- After: Single index seek using idx_characters_race_archetype
```

### 4. Statistics Queries

```sql
-- Before: Full table scan with date calculation
SELECT COUNT(*) FROM users WHERE createdAt >= ?;

-- After: Index range scan using idx_users_created_at_desc
```

## Monitoring and Maintenance

### Performance Monitoring

Monitor these metrics to validate optimization effectiveness:

- Query execution time reduction
- Index usage statistics
- Cache hit ratios
- Overall response time improvements

### Query Plan Analysis

Use `EXPLAIN ANALYZE` to verify indexes are being used:

```sql
EXPLAIN ANALYZE SELECT * FROM users
WHERE role = 'USER' AND isActive = true
ORDER BY createdAt DESC;
```

### Maintenance Tasks

1. **Regular ANALYZE**: Update table statistics monthly
2. **Index Usage Review**: Monitor unused indexes quarterly
3. **Performance Testing**: Benchmark critical queries after schema changes
